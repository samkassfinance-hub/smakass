# Migration Guide: Old → New Subscription System

This guide helps you migrate from the previous subscription system to the new **server-centric, tamper-proof** enforcement system.

---

## What's Changing

### Old System (Problems)
- ❌ localStorage stores subscription status (user can edit)
- ❌ Client-side expiry checks (unreliable)
- ❌ Client limit in JavaScript (easy to bypass)
- ❌ No UTC server time (clock manipulation possible)
- ❌ Duplicate subscription data in settings and localStorage

### New System (Fixed)
- ✅ Server database is single source of truth
- ✅ All expiry checks on backend (UTC time only)
- ✅ Client limit enforced on backend before adding
- ✅ Tamper-proof: no way to fake subscription status
- ✅ One canonical subscription record per user

## Step-by-Step Migration

### Step 1: Database Setup

#### Create Subscription Tables

In Supabase SQL Editor, run:

```sql
-- Copy-paste the entire SUBSCRIPTION_SCHEMA.sql file
-- It creates subscriptions and payment_history tables
```

Or manually create:

```sql
CREATE TABLE IF NOT EXISTS subscriptions (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  email TEXT NOT NULL UNIQUE,
  plan_type TEXT NOT NULL CHECK (plan_type IN ('free', 'oneday', 'monthly', 'quarterly', 'yearly')),
  plan_name TEXT NOT NULL,
  start_time TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  expiry_time TIMESTAMP WITH TIME ZONE,
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'expired', 'cancelled')),
  payment_id TEXT,
  razorpay_order_id TEXT,
  razorpay_signature TEXT,
  amount_paid DECIMAL(10, 2) DEFAULT 0,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

CREATE INDEX idx_subscriptions_email ON subscriptions(email);
CREATE INDEX idx_subscriptions_expiry_time ON subscriptions(expiry_time);
```

#### Migrate Existing Subscription Data (Optional)

If you have existing paid users, migrate their subscriptions:

```sql
-- This is a template. Adjust based on your old data structure.

INSERT INTO subscriptions (email, plan_type, plan_name, start_time, expiry_time, status, amount_paid)
SELECT 
  settings_email AS email,
  settings_plan AS plan_type,
  CASE settings_plan 
    WHEN 'monthly' THEN 'Monthly'
    WHEN 'quarterly' THEN 'Quarterly'
    WHEN 'yearly' THEN 'Yearly'
    ELSE 'Free Plan'
  END AS plan_name,
  settings_paymentDate AS start_time,
  settings_planExpiry AS expiry_time,
  CASE 
    WHEN settings_planExpiry > now() THEN 'active'
    ELSE 'expired'
  END AS status,
  0 AS amount_paid
FROM users_old
WHERE settings_plan IS NOT NULL AND settings_plan != 'free'
ON CONFLICT (email) DO NOTHING;
```

#### 2.3 Update Any Client-Adding Routes

Your existing client-adding route (e.g., `/api/clients/add`) must now verify subscription:

```python
from subscription_manager import subscription_manager

@app.route('/api/clients/add', methods=['POST'])
def add_client():
    user_email = request.headers.get('X-User-Email')
    data = request.get_json()
    
    # NEW: Check subscription
    subscription = subscription_manager.get_user_subscription(user_email)
    if subscription['is_expired'] and subscription['plan_type'] != 'free':
        return {'error': 'Subscription expired'}, 403
    
    # NEW: Check client limit
    client_count = subscription_manager.get_client_count(user_email)
    can_add, limit_info = subscription_manager.check_client_limit(user_email, client_count)
    
    if not can_add:
        return {
            'error': f'Client limit reached: {limit_info["limit"]}',
            'plan': limit_info['plan_name']
        }, 400
    
    # OLD: Rest of your existing logic
    # ... add client to database ...
```

#### 2.4 Update Razorpay Payment Verification

In your payment verification route, after successful Razorpay verification:

```python
from subscription_manager import subscription_manager

@app.route('/api/payment/verify', methods=['POST'])
def verify_payment_signature():
    data = request.get_json()
    user_email = data.get('email')
    plan_type = data.get('plan_type')
    razorpay_payment_id = data.get('razorpay_payment_id')
    razorpay_order_id = data.get('razorpay_order_id')
    razorpay_signature = data.get('razorpay_signature')
    amount_paid = data.get('amount_paid', 0)
    
    # Verify Razorpay signature (your existing code)
    is_valid = verify_razorpay_signature(...)  # Your existing verification
    
    if not is_valid:
        return {'error': 'Invalid signature'}, 400
    
    # NEW: Create subscription record
    success, result = subscription_manager.create_subscription(
        user_email=user_email,
        plan_type=plan_type,
        payment_id=razorpay_payment_id,
        razorpay_order_id=razorpay_order_id,
        razorpay_signature=razorpay_signature,
        amount_paid=amount_paid,
        start_time=datetime.utcnow()  # Exact payment confirmation time
    )
    
    if success:
        return {'success': True, 'subscription': result}, 200
    else:
        return {'error': result.get('error')}, 400
```

## Rollback Plan

If issues occur, you can temporarily roll back:

### Quick Rollback (Keep New Data)

1. Disable subscription enforcement:
   ```javascript
   // In app.js, comment out:
   // await window.SubscriptionEnforcement.initialize();
   ```

2. Restore old subscription checks if you kept them

3. Users revert to old behavior but keep new subscription data

### Full Rollback (Restore Old System)

1. Restore old code from git
2. Old subscription data in localStorage still works
3. Lose new subscription data (unless migrated back)

**Recommendation**: Keep old code available for 1-2 weeks as a safety net, then remove.

## Common Issues & Fixes

### Issue: "Module not found: subscription_manager"

**Fix**: Ensure `subscription_manager.py` is in `kaasflow/backend/` directory

```bash
ls kaasflow/backend/subscription_manager.py
```

### Issue: "Subscriptions table does not exist"

**Fix**: Run `SUBSCRIPTION_SCHEMA.sql` in Supabase SQL Editor

```sql
-- Copy entire SUBSCRIPTION_SCHEMA.sql file
-- Run in Supabase → SQL Editor
-- Verify in Supabase → Tables tab
```

### Issue: "User cannot add client after payment"

**Fix**: 
1. Check `/api/subscription/verify-payment` is called
2. Check subscription table has new record
3. Query: `SELECT * FROM subscriptions WHERE email = 'user@test.com'`
4. Check `plan_type` and `expiry_time` are correct

### Issue: "Blocking modal doesn't appear"

**Fix**:
1. Check browser console for errors
2. Verify `subscription-enforcement.js` is loaded
3. Check `window.SubscriptionEnforcement` exists
4. Verify `/api/subscription/status` returns `is_expired: true`

## Summary

**Before**: Subscription data in localStorage (can be edited by user)  
**After**: Subscription data on server (single source of truth)

**Old Flow**: 
- User pays → localStorage updated → UI checks localStorage → Access granted

**New Flow**:
- User pays → Server creates subscription → Frontend checks server → Server validates all API calls → Access granted

**Result**: Tamper-proof, server-enforced subscription system ✅
